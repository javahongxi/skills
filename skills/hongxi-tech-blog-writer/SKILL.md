# Tech Blog Writer

基于 javahongxi 5 篇技术博文的写作风格，提炼出一套可复用的技术博客写作规范。

## 触发方式

- "帮我写一篇推广博文"
- "写一篇技术博客介绍这个项目"
- "帮我写个升级公告博文"
- "写一篇项目深度解读"
- "write a tech blog post"

## 写作流程

### Step 1: 明确博文类型

与用户确认博文类型（4 种模板）：

| 类型 | 适用场景 | 代表作品 |
|------|---------|---------|
| **项目推广文** | 向社区推广开源项目 | cloud-promo-blog, whatsmars-promo-blog |
| **技术深度文** | 深入解读某项技术/模块 | spring-ai-blog |
| **项目全景文** | 项目整体概览与亮点 | cloud-blog |
| **升级公告文** | 新版本/新模块发布 | cloud-new-modules-blog |

### Step 2: 收集素材

向用户了解：
1. 项目名称、GitHub 地址
2. 核心技术栈与版本号
3. 模块列表与功能说明
4. 核心亮点（2-8 个）
5. 目标读者（初中级/高级/架构师）
6. 快速开始步骤

### Step 3: 按模板结构生成

严格遵循下方的**文章结构模板**。详细风格规范见 [style-reference.md](style-reference.md)。

## 各类型差异化要点

### 项目推广文
- 痛点部分着重"为什么需要"
- 亮点覆盖所有核心模块，每个模块 1-2 段
- 学习价值按 3 层受众分层
- 语气：全面、权威、有号召力

### 技术深度文
- 采用"层级递进"结构（第一层→第二层→...→第九层）
- 每层：概念引入 → 代码演示 → 效果展示 → 关键洞察
- 多用对比表格（temperature 对比、方案对比）
- 语气：深入浅出、有洞察力

### 项目全景文
- 用"亮点一/二/三..."编号
- 每个亮点：问题背景 → 实现方案 → 代码 → 意义
- 架构图必须详细（ASCII art）
- 模块速查表不可省略

### 升级公告文
- 开头承接上一篇博文（"上一篇博文中..."）
- 新增模块用 🆕 标注
- 每个新模块独立成章，包含完整的使用示例
- 末尾"模块总览"表格标注新旧状态

## 写作规范速查

| 维度 | 规范 |
|------|------|
| 语言 | 中文为主，技术术语保留英文 |
| 段落 | 短段落（2-4 行），多留白 |
| 强调 | **加粗**用于核心观点和结论 |
| 代码 | 每个亮点至少 1 个代码示例 |
| 表格 | 模块对比、功能矩阵用表格 |
| 架构图 | ASCII art，不用外部图片 |
| Emoji | 仅用于章节标题，正文不用 |
| 签名 | `© [hongxi.org](http://hongxi.org)` |
| 篇幅 | 推广文 300-550 行，深度文 350-450 行 |

## 禁忌

- ❌ 不用"本文将..."开头（用痛点引入）
- ❌ 不在正文中使用 emoji（仅标题）
- ❌ 不写空洞的形容词（"非常好""很棒"）→ 用数据/代码说话
- ❌ 不省略代码示例（每个亮点必须有代码）
- ❌ 不省略架构图（ASCII art 是标配）
- ❌ 不写没有结论的章节（每个亮点以"这意味着：**...**"收尾）
