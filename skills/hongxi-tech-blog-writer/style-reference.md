# 技术博文风格参考手册

基于 javahongxi 5 篇博文的详细风格分析，提供可直接套用的写作范式。

## 一、标题写作范式

### 主标题公式

**公式 1：数字 + 卖点**
```
Spring Cloud Alibaba 生产级实战：16 个模块覆盖全栈微服务
```

**公式 2：场景 + 技术栈**
```
当 Java 开发者遇上 Spring AI 2.0：从 Chat 到 Agent，一文打通 AI 全栈能力
```

**公式 3：身份 + 全景图**
```
一个 Java 后端工程师的技术全景图：whatsmars 项目深度解读
```

**公式 4：升级 + 新模块**
```
Spring Cloud Samples 大升级：AI 智能体 + gRPC + Seata 分布式事务，三大重磅模块来袭
```

### 副标题（blockquote）

**格式**：用 `|` 分隔 3-5 个技术标签
```markdown
> **Spring Boot 4.x** | **Spring Cloud Alibaba 2025.1.x** | **Dubbo 3.3.6**
> **Spring Boot 4.1** | **Spring AI 2.0** | **Spring gRPC** | **Seata 分布式事务** | **MCP 协议**
```

**或一句话定位**：
```markdown
> 不需要 Python，不需要学新框架。用你熟悉的 Spring Boot，3 行代码接入大模型，10 行代码构建 AI Agent。
```

## 二、开头引入范式

### 痛点共鸣式（最常用）

```markdown
在微服务架构日益普及的今天，如何快速上手 Spring Cloud Alibaba 生态？如何确保你的技术选型与最新框架版本兼容？本文将为你介绍一个精心打造的 Spring Cloud Alibaba 示例项目...
```

**结构**：[行业背景] + [痛点问题] + [项目引出]

### 对比反差式

```markdown
你可能已经见过太多 AI 演示——一个聊天框，输入问题，输出答案。然后呢？

**我们要做的是完全不同的事情。**
```

**结构**：[常见现象] + [转折] + [我们的方案]

### 直接痛点列表

```markdown
开发者在学习时经常遇到这些问题：
- 看了一堆文档，不知道怎么串起来
- 示例代码太简单，无法参考到生产级实践
- 环境搭建复杂，跑通一个 Demo 要花半天
```

**结构**：[引导句] + [3 个 bullet point 痛点]

## 三、章节标题 Emoji 规范

### 常用 Emoji 映射

| Emoji | 用途 | 示例 |
|-------|------|------|
| 🎯 | 目标/动机 | 🎯 为什么需要这个项目？ |
| 🚀 | 亮点/发布 | 🚀 项目亮点 |
| 🏗️ | 架构/设计 | 🏛️ 架构设计 |
| 🏛️ | 架构（变体） | 🏛️ 架构设计哲学 |
| 📊 | 数据/表格 | 📊 模块速查表 |
| 🎓 | 学习/教育 | 🎓 学习价值 |
| 🚦 | 开始/启动 | 🚦 快速开始 |
| 💡 | 实践/技巧 | 💡 最佳实践总结 |
| 🔗 | 链接 | 🔗 相关链接 |
| 📝 | 总结/结语 | 📝 结语 |
| 🤝 | 贡献 | 🤝 贡献指南 |
| ✨ | 具体亮点 | ✨ 1. 紧跟技术前沿 |
| 📦 | 项目地址 | 📦 **项目地址**: [URL] |
| 📖 | 文档 | 📖 **Spring Boot**: [URL] |

### 亮点编号格式

```markdown
## 亮点一：gRPC 服务发现——桥接 Spring Cloud 与 gRPC 两套体系
## ✨ 1. 紧跟技术前沿
### 🤖 模块一：cloud-ai-sample —— Spring AI 2.0 全场景实战
```

**规则**：
- 大亮点用中文数字（一、二、三）或阿拉伯数字（1. 2. 3.）
- 模块用 emoji + 编号（🤖 模块一）
- 子亮点用 ### 三级标题

## 四、代码示例规范

### 代码前必有引导句

```markdown
核心代码仅一行 —— 通过 `entity()` 方法将 AI 响应直接映射为 Java record：

```java
return chatClient.prompt(new Prompt(prompt))
        .call()
        .entity(PersonInfo.class);
```
```

### 代码后必有结论句

```markdown
这意味着：**gRPC 服务可以像 Feign 调用一样，通过注册中心自动发现后端实例，无需额外部署 Consul DNS 或修改 gRPC 源码。**
```

### 多段代码用表格对比

```markdown
| 能力 | 接口 | 亮点 |
|------|------|------|
| 普通对话 | `/ai/chat` | 同步响应 |
| 流式输出 | `/ai/chat/stream` | SSE 实时推送 |
| 结构化输出 | `/ai/extract` | 自然语言 → Java Record |
```

### bash 命令示例

```markdown
```bash
# 1. Clone 项目
git clone https://github.com/javahongxi/spring-cloud-samples.git
cd spring-cloud-samples

# 2. 确保 Nacos 已运行（没有？告诉 AI "安装 Nacos"）

# 3. 告诉 AI "演示本项目"
# 或者手动启动：
sh start-all.sh
```
```

**规则**：每步都有注释说明

## 六、ASCII 架构图规范

### 微服务架构图

```
                          ┌─────────────┐
                          │   Nacos     │
                          │  注册/配置   │
                          └──────┬──────┘
                                 │
    ┌────────────────────────────┼────────────────────────────┐
    │                            │                            │
    │    ┌───────────┐    ┌──────┴──────┐    ┌───────────┐   │
    │    │  Gateway   │    │  Sentinel   │    │   Seata   │   │
    │    │   (8764)   │    │   限流规则   │    │  Server   │   │
    │    └─────┬─────┘    └─────────────┘    └─────┬─────┘   │
```

**规则**：
- 用 `┌─┐│└─┘` 等制表符
- 标注端口号 `(:8080)`
- 标注核心功能 `（注册/配置）`
- 层级清晰，从上到下

### 模块关系图

```
cloud-sample-api (proto 定义)
    └── greeter.proto ──▶ GreeterGrpc / GreeterRequest / GreeterReply

cloud-commons (公共库)
    └── DiscoveryClientNameResolverProvider  ← 识别 discovery:/// scheme
    └── DiscoveryClientNameResolver          ← 调用 DiscoveryClient 解析实例
```

**规则**：
- 用 `──▶` 表示依赖方向
- 用 `←` 表示反向引用
- 模块名 + 一句话说明

## 七、学习价值分层范式

### 三层受众模型

```markdown
## 🎓 学习价值

### 对于初学者 / 初中级开发者
- ✅ 从零搭建完整的微服务项目
- ✅ 理解服务注册发现、配置中心、网关、RPC 等核心概念
- ✅ 掌握 Spring Cloud Alibaba 各组件协作方式

### 对于进阶开发者 / 高级开发者
- ✅ Spring Boot 4.0 新特性实践（Tracing、Observation API）
- ✅ Dubbo 3.x 与 Spring Cloud 集成最佳实践
- ✅ WebFlux 响应式编程模型

### 对于架构师
- ✅ 多协议服务治理方案（HTTP + Dubbo）
- ✅ 微服务可观测性建设（Tracing + Metrics）
- ✅ 生产环境部署脚本设计
```

**规则**：
- 每层 3-5 个 ✅ 要点
- 从基础到高级递进
- 用具体技术名词，不用空泛描述

## 九、语气与措辞规范

### 自信但不夸张

✅ **推荐**：
- "这不是一个简单的 Demo 集合，而是一个**面向生产环境的微服务参考架构**"
- "Spring AI 2.0 让 Java 开发者用熟悉的方式拥抱 AI。不是'也能用'，而是**用得优雅**"

❌ **避免**：
- "这是世界上最好的项目"
- "绝对完美无缺"

### 用数据说话

✅ **推荐**：
- "16 个模块、覆盖 10+ 核心组件、提供一键演示脚本"
- "从 clone 到跑通，最快 30 秒"

❌ **避免**：
- "很多模块""很快""非常全面"

### 用对比制造冲击

✅ **推荐**：
- "从 3 行代码的简单对话，到自主调用 11 个工具的 ReAct Agent"
- "从'AI 会胡说八道'到 RAG 知识约束"
- "从无状态的记忆丢失到 JDBC 持久化的会话隔离"

❌ **避免**：
- "我们的 AI 功能很强大"

### 用反问制造共鸣

✅ **推荐**：
- "在微服务架构日益普及的今天，如何快速上手 Spring Cloud Alibaba 生态？"
- "为什么覆盖四种？因为不同的业务场景需要不同的消息中间件"

## 十、完整检查清单

生成博文后，逐项检查：

### 结构完整性
- [ ] 主标题 + 副标题（blockquote）
- [ ] 痛点引入（至少 2 个痛点）
- [ ] 亮点/模块章节（每个有代码示例）
- [ ] 架构图（ASCII art）
- [ ] 模块速查表（表格）
- [ ] 学习价值（三层受众）
- [ ] 快速开始（前置要求 + 步骤）
- [ ] 最佳实践（至少 2 条）
- [ ] 相关链接（5-8 个）
- [ ] 结语 + 号召 + 签名

### 风格一致性
- [ ] 章节标题使用 emoji
- [ ] 核心观点**加粗**
- [ ] 每个亮点有"这意味着：**结论**"
- [ ] 代码块前有引导句
- [ ] 代码块后有结论句
- [ ] 表格对齐整齐
- [ ] 架构图用制表符
- [ ] 短段落（2-4 行）
- [ ] 无空洞形容词

### 技术准确性
- [ ] 版本号准确
- [ ] 端口号标注
- [ ] 代码可运行
- [ ] 链接有效
- [ ] 模块名拼写正确

### 签名格式
- [ ] `© [hongxi.org](http://hongxi.org)`
- [ ] 可选：`| 一句话 slogan`
